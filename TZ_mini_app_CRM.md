# ТЗ: Мини-CRM для MatchMatch (Telegram Mini App)

> Версия 1.0. Управление лидами WhatsApp-бота Anna прямо из Telegram, без текстовых команд.

---

## 1. Зачем это нужно

Сейчас Аня управляет ботом через текстовые команды в Telegram (`/leads`, `/lead <номер>`, `/takeover` и т.д.). Это работает, но неудобно при реальном объёме трафика:
- Команды надо помнить и набирать руками
- Номера телефонов копировать вручную
- Нет общей картины (сколько лидов, на какой стадии, статистика)
- Нет удобного способа выгрузить данные

Цель — веб-интерфейс, открывающийся прямо в Telegram (Telegram Mini App), который даёт то же самое управление, но кнопками и таблицами вместо команд.

**Важно:** это НЕ отдельный бэкенд-проект. Вся бизнес-логика (база данных, статусы, фильтры, воронка) уже готова и работает в основном боте (13 блоков написаны и задеплоены). Мини-апп — это **фронтенд поверх существующих данных**, читает/пишет в ту же Supabase-базу через новый набор API-эндпоинтов.

---

## 2. Кто пользуется

- **Аня** (основной пользователь) — владелец агентства, видит всех лидов, управляет воронкой
- **Разработка** (Arina) — техническая поддержка, тот же доступ для отладки

Доступ ограничен по Telegram user_id (та же логика авторизации что уже есть в menedžer-боте — `TG_MANAGER_ADMIN_IDS`).

## 3. Функциональные требования (что должно быть)

### 3.1 Список лидов (главный экран)
- Таблица/список всех лидов с колонками: имя, телефон, стадия воронки, последнее сообщение (дата+превью), режим (бот/вручную), пометка если в списке клиентов
- **Фильтры**: по стадии воронки, по режиму (бот отвечает / вручную), по интересу (event/agency), по дате последнего сообщения
- **Поиск**: по имени или телефону
- **Сортировка**: по дате последнего сообщения (по умолчанию — свежие сверху), по стадии
- Пагинация или бесконечная прокрутка (лидов может быть много)

### 3.2 Карточка лида
- Полная информация: имя, телефон, возраст, профессия, город, статус (soltero и т.д.), интерес, стадия воронки, режим, дата первого/последнего сообщения
- Вся история переписки (как в боте — кто написал: лид / Anna / менеджер вручную)
- Пометка если в списке "клиенты" (whitelist) — с причиной и кем добавлен
- **Действия прямо с карточки**:
  - Взять переписку себе (takeover) / Вернуть боту (release) — переключатель, без набора команд
  - Добавить в клиенты / убрать из клиентов
  - Остановить бота для этого лида (аналог текущего /stop)
  - Открыть переписку в WhatsApp (кнопка-ссылка)
  - Написать сообщение вручную прямо из интерфейса (отправка через бота от имени менеджера)
- Фото лида (если есть) с вердиктом Vision-проверки

### 3.3 Управление списком "клиентов" (whitelist)
- Отдельный экран: список всех кто в whitelist (имя, телефон, причина, кто добавил, дата)
- Добавить нового — по номеру телефона (с валидацией формата) + причина
- Убрать — одним кликом
- Поиск/фильтр по этому списку

### 3.4 Экспорт
- Кнопка "Скачать таблицу" — выгружает текущий отфильтрованный список лидов в Excel/CSV
- Поля: все основные (имя, телефон, возраст, профессия, город, стадия, интерес, даты, режим)

### 3.5 Дашборд/статистика (не приоритет 1, но заложить архитектурно)
- Количество лидов по стадиям воронки (простая диаграмма/цифры)
- Сколько новых за сегодня/неделю
- Сколько эскалаций ждут ответа

### 3.6 Настройки ивента (перенос текущих команд в интерфейс)
- Форма: дата, время, адрес ивента
- Загрузка/указание картинки-приглашения + переключатель "отправлять"
- Ссылка на оплату/бронь ивента
- Ссылка на курсы (course_link)
- Всё то же что сейчас команды `/set_event`, `/set_invitation`, `/set_event_link`, `/set_course_link` — но полями формы

### 3.7 Уведомления (опционально, фаза 2)
- Внутри мини-аппа видно то же что сейчас приходит алертами в Telegram (эскалации, VIP-сообщения, фото на ручной проверке) — как единый "инбокс" вместо отдельных сообщений в чате

## 4. UX/UI принципы

### 4.1 Общий стиль
- Чисто, минималистично, без перегруза — Аня не технарь, интерфейс должен считываться сразу
- Тёплая цветовая гамма в духе бренда MatchMatch (можно взять акцентный цвет из уже существующих материалов агентства, если есть брендбук/сайт — сверить с matchmatchagency.com)
- Мобильный интерфейс в приоритете (Telegram Mini App чаще открывается с телефона), но должен нормально работать и на десктопе Telegram

### 4.2 Информационная архитектура (структура экранов)
```
Главный экран (список лидов)
├── Карточка лида (клик на строку)
│   ├── История переписки
│   ├── Кнопки действий (takeover/release/клиенты/стоп)
│   └── Написать вручную
├── Клиенты (отдельная вкладка/экран)
│   └── Добавить/убрать
├── Настройки ивента (отдельная вкладка/экран)
└── Экспорт (кнопка в шапке главного экрана)
```

### 4.3 Ключевые UX-принципы
- **Меньше кликов до действия** — takeover/release должны быть одним тапом с карточки лида, не через 3 экрана
- **Понятные статусы без жаргона** — те же принципы что уже применили в текстовом боте: без слов "заблокирован", "whitelist" — только "Клиент из списка", "Бот прекратил диалог" и т.п. (терминология уже отработана в блоке 11, переносим её же)
- **Визуальные индикаторы стадии воронки** — цветовые метки/бейджи по стадии (например new — серый, qualified — синий, client_* — зелёный, rejected/lost — красный/приглушённый), чтобы Аня могла быстро сканировать список глазами
- **Живое обновление** (если технически несложно) — новые сообщения/эскалации видны без перезагрузки страницы, или хотя бы кнопка "обновить"

## 5. Технический план

### 5.1 Технология
**Telegram Mini App** — обычное веб-приложение (React/Vue/или проще — vanilla + библиотека), встроенное в Telegram через Telegram Web App SDK. Открывается по кнопке в менеджер-боте.

### 5.2 Архитектура (переиспользование)
```
Существующий бэкенд (main.py, db.py, всё что уже есть)
        │
        ├── Существующий Telegram-бот (команды, алерты) — остаётся как есть
        │
        └── НОВОЕ: API-эндпоинты для мини-аппа
                (/api/leads, /api/lead/<phone>, /api/whitelist, /api/export, /api/event-settings)
                — тонкий слой поверх существующих db-функций (list_active_leads,
                get_lead_by_phone, add_to_whitelist, set_manual/set_auto и т.д.)
                        │
                        └── Веб-фронтенд (отдельная папка/проект)
                                читает API, рендерит интерфейс
```

Авторизация — через Telegram initData (стандартный механизм Mini Apps, проверяет подпись что открыл именно авторизованный пользователь Telegram, та же проверка admin_ids что уже есть).

### 5.3 Хостинг
Фронтенд — статика (можно на том же сервере через nginx, который уже настроен для бота, или отдельно на Vercel/Netlify — проще для итераций дизайна).
Бэкенд API — новые роуты в существующем FastAPI-приложении (тот же сервер, тот же процесс).

## 6. Приоритеты (что сначала)

**Фаза 1 (MVP, самое нужное сейчас):**
- Список лидов с фильтрами/поиском
- Карточка лида с историей + действия (takeover/release/клиенты/стоп)
- Экспорт в таблицу
- Управление списком клиентов

**Фаза 2 (после MVP, когда будет понятно что ещё нужно):**
- Настройки ивента через форму
- Дашборд/статистика
- Единый инбокс уведомлений
- Отправка сообщений вручную из интерфейса

## 8. Не входит в это ТЗ (сознательно исключено)

- Полная замена Telegram-алертов мини-аппом — алерты остаются, мини-апп дополняет
- AI-функции внутри мини-аппа — это отдельная история (см. AI-follow-up, обсуждается отдельно)
- Управление контентом сценариев бота через интерфейс — пока через БД/код напрямую
